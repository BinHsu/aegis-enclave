# ADR-0042: Data store — DynamoDB Global Tables (active-active multi-region)

## Status
Accepted (2026-04-28)

## Context

aegis-enclave is a **Tier 2 ops support workload** (per ADR-0008). The data layer requirements:

| Workload property | Shape | Implication |
|---|---|---|
| Write pattern | Insert-mostly (audit row per request); rare update for status transitions (queued → running → done) | NoSQL audit-log shape; no JOIN-heavy queries |
| Read pattern | By `execution_id` (poll endpoint); occasional time-range scan (worker stale-running detection); no transactional cross-row reads | Single-table model with optional GSI on time/status |
| Foreign keys | None load-bearing (audit table is append-only; no tenant/user FK to enforce) | DynamoDB single-table fits |
| ACID requirements | Single-item conditional writes for idempotency | DynamoDB conditional expressions native support this |
| PII / regulated data | None for the prime-computation workload | No GDPR-driven retention complexity beyond audit-log baseline |
| Scale curve | Burst 50–100 RPS for ≤ 30 s, idle baseline ~1 req/min | Both DynamoDB on-demand and provisioned modes fit; on-demand simpler |

This is a textbook **cloud-native NoSQL audit log** workload. The relational features (JOIN, multi-row transactions, schema migration discipline) that SQL provides are not features the workload uses.

### Multi-region rationale

Per ADR-0007 (per-region 3-AZ posture) + ADR-0008 (Tier 2 RTO ≤ 15 min): single-region 3-AZ is sufficient for the SLO target. Multi-region is a quality-of-engineering signal beyond Tier 2 baseline, not a compliance mandate for this workload class.

The choice within multi-region is **active-active over active-passive**:

- DynamoDB Global Tables is multi-master by design — every region accepts writes natively, with last-writer-wins conflict resolution at cell level and ~1 s cross-region replication lag.
- Operating it active-passive (one region as writer, other as standby) underutilises the multi-master capability. Cost is identical (both regions provisioned in either case); only routing changes.
- Active-active distributes load naturally and gives closer-region operators lower latency.
- "Failover" reduces to "Route53 weighted record from 50/50 to 100/0" — no promotion step, no failback reconstitution.

## Decision

### Data layer

- `aws_dynamodb_table.executions` with `replica` configurations for `eu-central-1` and `eu-west-1`
- **Primary key**: `execution_id` (String, UUID4 generated by API on insert) — UUIDs avoid hot-partition issues that integer-sequence keys would create
- **Sort key**: none (single-item-per-PK model)
- **Attributes**: `status`, `range_start`, `range_end`, `primes` (List), `primes_count`, `duration_ms`, `error_message`, `created_at` (Number, epoch seconds), `started_at`, `completed_at`, `ttl_at` (Number, epoch seconds for DynamoDB TTL feature)
- **GSI**: none in v1. If ops need "list all running executions for stale-detection at scale" later, add `status_index` (PK = `status`, SK = `created_at`). For current scale, scan with FilterExpression is acceptable.
- **TTL**: status-aware via the `ttl_at` attribute set on insert/update:
  - `status = done` → TTL = created_at + 30 days
  - `status = failed` → TTL = created_at + 90 days (forensic / RCA support)
  - `status = running` or `queued` → no TTL (don't auto-delete in-flight work)
- **Capacity mode**: on-demand (PAY_PER_REQUEST) — burst-shaped workload doesn't justify provisioned reservation
- **Stream**: `NEW_AND_OLD_IMAGES` enabled (V2 hook for archival to S3 / event-driven downstream consumers)

### Read/write semantics

- **Idempotency check** (worker before computing): `update_item` with `ConditionExpression: status = queued AND attribute_exists(execution_id)` — atomic, no race conditions.
- **Status transitions** (queued → running → done | failed): conditional `update_item` with `ConditionExpression` checking previous status. Prevents out-of-order updates.
- **Local read** (within originating region): `ConsistentRead=true` for strong-consistent reads on the local replica. Sub-millisecond.
- **Cross-region read** (operator polls a region different from the writer): eventual consistency (~1 s replication lag). Acceptable for the polling pattern; client polls again if status hasn't propagated. RPO = 1 s.
- **Conflict resolution** (concurrent writes from different regions to same item): last-writer-wins by writer-side timestamp. UUID4 PK makes cross-region collision statistically zero; status transitions are monotonic so concurrent updates converge.

### Routing layer (Route53)

- Single hosted zone for the deployment domain (e.g., `api.enclave.example`)
- Two record sets, **weighted routing policy** 50/50 between Frankfurt ALB and Ireland ALB
- Each record set has a **health check** on the ALB's `/health` endpoint
- TTL = 60 s on records (cap client-side DNS cache)
- Health check 3 consecutive failures (90 s) → Route53 stops returning that record; clients re-resolve on TTL expiry

### Compute layer (per region)

- Each region has its own VPC + ALB + ECS cluster + Valkey + SQS + Client VPN endpoint (mirror structure per ADR-0007)
- ECS app + worker services run at the same `desired_count` baseline in both regions (active-active)
- Worker reads from + writes to the same region's DynamoDB Global Tables replica (local strong-consistent reads + writes)
- SQS messages are region-local (no cross-region SQS bridging; messages stay in originating region for processing locality)
- Valkey cache is region-local (no cross-region cache replication; bootstrap re-seeds in each region independently)

### Failover semantics

- **No promotion step.** Both regions are writers always.
- **No Lambda automation.** Route53 health check + record TTL handle traffic shift.
- **No failback complexity.** When a failed region recovers, its health check returns OK, Route53 restarts returning its record, and the in-region replica catches up via Global Tables replication automatically.
- **RTO breakdown**:
  - Route53 health check 3 consecutive failures × 30 s = 90 s
  - DNS TTL = 60 s; client cached resolutions expire over 60–300 s
  - In-flight requests on failed region: ALB returns 5xx; client retries hit healthy region after DNS refresh
  - **Total RTO ≈ 60–300 s** (DNS-dominated; ECS already running on healthy region, no scale-up needed)
- **RPO**: ~1 s typical Global Tables cross-region replication lag.

### Local development parity

- `docker-compose.yml` uses `amazon/dynamodb-local` mirroring ADR-0030's ElasticMQ-for-SQS pattern
- Smoke test exercises against `http://dynamodb-local:8000`; production-shape connection via `boto3.client("dynamodb", endpoint_url=...)` toggles between local/cloud via env var
- Python unit tests use `moto[dynamodb]` in-process for speed
- Multi-region semantics are NOT locally emulated (`amazon/dynamodb-local` doesn't support Global Tables) — verified during cloud-up window only

## Alternatives Considered

Industry-pattern alternatives, sparse:

- **Aurora Global Database** — alternative for organisations with existing PostgreSQL investment. Active-passive with Lambda-driven failover; failback Path 1 / Path 2 reconstitution complexity (1–12 h on disaster path). Right call when PG schema + tooling represent prior commitment cost; otherwise the failover-orchestration tax is not justified.
- **Vanilla RDS PG with cross-region read replica** — Tier 0/1 active-passive option; manual `aws rds promote-read-replica` failover; RPO measured in minutes.
- **CockroachDB / YugabyteDB** — distributed SQL alternatives; PostgreSQL wire-protocol compatible (CockroachDB). Right call for orgs with platform-team capacity to operate distributed databases; over-built for this workload size.
- **CockroachDB Cloud** (managed) — same architectural shape, vendor-managed. Right call for cloud-agnostic deployments; introduces non-AWS dependency.
- **DynamoDB single-region + cross-region snapshot** — cheaper than Global Tables; RPO becomes hours, restore from snapshot is hours. Suitable for V3 disaster-recovery posture, dev/test environments — not active-passive multi-region production.
- **DynamoDB Global Tables active-passive** — underutilises multi-master capability; secondary cost is identical, routing-weight 50/50 vs 100/0 is one terraform line. Active-active is the architecturally honest choice when the storage layer is multi-master.
- **Active-active with two independent DDB tables + custom application replication** — re-implements multi-master at app layer; Global Tables solves this at storage layer.

## Consequences

- **Idempotent writes**: DDB conditional expressions replace SQL `INSERT ON CONFLICT IGNORE` semantics; same correctness, no transaction.
- **No schema migration step on deploy**: DDB tables are terraform-managed; the bootstrap task narrows to cache pre-warm only.
- **Local parity**: `amazon/dynamodb-local` Docker image is the local data layer (mirrors ADR-0030's ElasticMQ-for-SQS pattern). `docker-compose.yml` includes it.
- **Test layer**: `moto[dynamodb]` in unit tests for speed.
- **SLI emission unchanged**: `src/prime_service/metrics.py` EMF emission is data-layer-agnostic; all metrics in ADR-0041 remain the same.
- **Cost shape (eu-central-1 + eu-west-1)**: roughly 2× single-region per-hour cost (~$1.68/h vs $0.84/h per-region; per-region cost per ADR-0007 3-AZ posture). Plus DynamoDB Global Tables billing: replicated writes at 1× rWCU on PAY_PER_REQUEST (~$1.875 per million writes). For burst-sized workload (~10 k requests/day): ~$0.10–0.30/month replication cost, trivial.
- **Operator UX simplification**: no Lambda function to maintain; no failover runbook complexity; no "which region is primary right now?" mental overhead. Active-active means both regions are always primary for writes; Route53 weight is the only routing variable.
- **`tfvars-init.sh` prompts** for `primary_region` (default eu-central-1) + `secondary_region` (default eu-west-1). Single-region case-study scope can leave `secondary_region = ""` to disable Global Tables provisioning.
